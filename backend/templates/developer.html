Core Concept: Three-State Protocol

Every AtomicPay transaction exists in exactly one of three states. No transaction may remain in state 0.

-1 Tamas (Reversed) ← 0 Rajas (Transit) → +1 Sattva (Completed)

The zero state exists only during the atomic transit window (target: under 500ms). If any failure occurs, the transaction automatically snaps to -1. This eliminates stuck transactions and suspense accounts by design.

API Reference

All endpoints use JSON. Authenticated endpoints require Bearer token in the Authorization header.

Authentication

POST /api/register
Register a new user. Body: { name, mobile, pin }
POST /api/login
Login and receive JWT token. Body: { mobile, pin }

Banking

GET /api/banks
List available banks and their online status
POST /api/link_bank
Link bank account. Auth required. Body: { bank_id, account_id }
GET /api/balance
Get account balance from linked bank. Auth required.

Payments

POST /api/pay
Execute atomic payment with two-phase commit. Auth required. Body: { receiver_mobile, amount, note }. Payment requires PIN re-authentication on the client side.
POST /api/find_user
Find a user by mobile number. Auth required. Body: { mobile }
GET /api/history
Get transaction history (last 50). Auth required.

KYC

POST /api/kyc/submit
Submit KYC document. Auth required. Body: { document_type: "PAN"|"AADHAAR"|"PASSPORT", document_number }
GET /api/kyc/status
Get KYC verification status and submitted documents. Auth required.

Regulatory (Admin only)

GET /api/regulatory/reports
RBI sandbox compliance report with system metrics.
GET /api/regulatory/audit_logs
Full audit trail for regulatory review.
GET /api/regulatory/transactions
Complete transaction data for regulatory analysis.

Bank Onboarding Guide

Real banks can integrate with AtomicPay in three steps. This guide covers everything from registration to going live.

Step 1: Register Your Bank

Contact AtomicPay admin or use the Admin Dashboard → Bank Onboarding tab to register. You'll receive:

API Key: apk_live_a1b2c3d4e5f6... (authenticate your requests) Webhook Secret: whsec_x9y8z7w6v5... (verify webhook signatures) Status: pending (admin approval required)

Step 2: Implement the Bank Bridge API

Your bank must expose a single HTTPS POST endpoint that handles four actions. AtomicPay will call this endpoint during payment processing.

Endpoint: POST https://your-bank-api.com/atomicpay Headers: Content-Type: application/json X-AtomicPay-Key: <your_api_key> Actions your endpoint must handle: 1. DEBIT - Lock/debit funds from sender account Request: {"action": "DEBIT", "tx_id": "uuid", "account_id": "ACC_001", "amount": 500.00} Success: {"state": 1, "reason": "DEBITED", "name": "Account Holder", "new_balance": 9500.00} Failure: {"state": -1, "reason": "INSUFFICIENT_FUNDS", "available": 200.00} 2. CREDIT - Credit funds to receiver account Request: {"action": "CREDIT", "tx_id": "uuid", "account_id": "ACC_002", "amount": 500.00} Success: {"state": 1, "reason": "CREDITED", "name": "Account Holder", "new_balance": 5500.00} Failure: {"state": -1, "reason": "ACCOUNT_NOT_FOUND"} 3. UNLOCK - Rollback a debit (called if credit fails) Request: {"action": "UNLOCK", "tx_id": "uuid", "account_id": "ACC_001", "amount": 500.00} Success: {"state": 1, "reason": "UNLOCKED"} 4. STATUS - Verify account exists and check balance Request: {"action": "STATUS", "account_id": "ACC_001"} Success: {"state": 1, "name": "Account Holder", "balance": 10000.00, "active": true} Failure: {"state": -1, "reason": "NOT_FOUND"}

Step 3: Testing and Approval

Once registered, the AtomicPay admin will:

1. Run automated connectivity tests (ping, STATUS, DEBIT actions) 2. Verify your endpoint responds correctly for all 4 actions 3. Test a sandbox payment end-to-end 4. Approve the bank to go active 5. Your bank appears in the available banks list for users to link

Error Codes Reference

state: -1 Responses (Failure) ACCOUNT_NOT_FOUND - Account ID does not exist ACCOUNT_FROZEN - Account is suspended/frozen INSUFFICIENT_FUNDS - Balance too low for debit INVALID_AMOUNT - Amount is zero or negative INTERNAL_ERROR - Server-side processing error state: 1 Responses (Success) DEBITED - Funds successfully held/debited CREDITED - Funds successfully credited UNLOCKED - Debit successfully reversed

Webhook Signature Verification

Verify that requests are genuinely from AtomicPay using your webhook secret:

import hmac, hashlib def verify_signature(webhook_secret, payload_body, received_signature): expected = hmac.new( webhook_secret.encode(), payload_body.encode(), hashlib.sha256 ).hexdigest() return hmac.compare_digest(expected, received_signature)

Payment Flow

1. Client sends POST /api/pay { receiver_mobile, amount, note } 2. Client-side PIN/biometric verification required 3. Gateway creates transaction (state = 0, Rajas) 4. Gateway calls sender bank: DEBIT 5. Gateway calls receiver bank: CREDIT 6a. Both succeed → state = +1 (Sattva) → COMPLETED 6b. Any fails → UNLOCK sender → state = -1 (Tamas) → REVERSED 7. Transaction signed with HMAC-SHA256 8. Fraud score calculated, AML rules checked 9. Full audit trail recorded

Security

AtomicPay implements production-grade security measures for fintech compliance:

Authentication: JWT tokens with PBKDF2-SHA256 PIN hashing Authorization: Server-side role verification on all admin endpoints Rate Limiting: 5 login attempts per 5-minute window PIN Confirmation: Required before every payment transaction Biometric: WebAuthn fingerprint support for payment auth Signatures: HMAC-SHA256 on every transaction record Fraud: Real-time scoring (velocity, volume, KYC level) AML: Structuring detection, rapid transfer monitoring KYC: 3-level verification (PAN, Aadhaar, Passport) Audit: Complete event trail for regulatory compliance UPI Compatible: Interoperable with GPay, PhonePe, Paytm
